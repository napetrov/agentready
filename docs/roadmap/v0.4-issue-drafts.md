These are ready-to-open GitHub issue drafts for the next milestone, sourced
from the first real calibration record
([`reports/evaluation/calibration/napetrov-AIReady.json`](../../reports/evaluation/calibration/napetrov-AIReady.json))

## Milestone: v0.4 — Autonomy envelope and calibration feedback loop

Goal: move AgentReady from answering "is this repository ready for agents?"
to "what level of agent autonomy is this repository ready for?", and make the
gap between a scan's score and a deeper expert review a routine, structured
input instead of a one-off narrative.

The motivating case: `napetrov/AIReady` is not an obviously weak repository —
extensive agent instructions, sophisticated CI, and a dedicated AgentReady
Action gate that passed at a configured minimum score of 80. A manual review
still found a stale quick-start command, documentation drift between
playbooks/scripts and the current implementation, an automatic session hook
that executes branch-controlled installation code, thin high-risk-path
ownership, and unverifiable branch-protection state. None of that means the
scan was wrong — it means a single aggregate score cannot communicate what
those gaps mean for autonomy, and some of them are gaps in coverage, not
scoring.

Done when:

- every finding carries `affectedStage` metadata and a report can render an
  autonomy-envelope summary from it — **done**, see Issue 4
- the calibration feedback schema and format are documented and have at least
  one real record — **done**, see Issue 1
- at least one net-new deterministic check and one policy option closes a
  `false_negative`/`policy_mismatch` from the AIReady record — **done**, see
  Issues 2, 3, 6, 7
- reports distinguish "found nothing" from "cannot verify this locally" —
  **done**, see Issue 5

**Status: all seven issues below are delivered.** Only the three items under
"Candidates not scoped for v0.4" remain open.

### Acceptance criteria

## Issue 2: Package-manager-aware shortcut-script command validation

Labels: `enhancement`, `detector`, `v0.4`

**Status: delivered**, with one deliberate deviation from the scope below.
`commands.reference.shortcut-script` (`lib/repo-readiness/detectors/command-references.ts`)
ships as a single `warning`-severity rule (matching `npm-script`/`make-target`)
rather than a two-tier error/info confidence split. Instead of a confidence
tier, precision comes from requiring the reference to sit inside a Markdown
code span (inline `` ` `` or fenced ```` ``` ````) — a real "run this" doc
instruction is almost always shown as code, while prose that merely mentions
"the pnpm ecosystem" is not. This solves the same false-positive problem the
two-tier design was reaching for with one mechanism instead of two, and every
example in the existing test suite (and this repo's own docs) already uses
code-formatted commands. The `oss` policy pack escalates it to `error`
alongside the other two stale command-reference kinds.

### Problem

`detectCommandReferences`
(`lib/repo-readiness/detectors/command-references.ts`) intentionally only
checks explicit `<pm> run <script>` and the two safe bare forms (`test`,
`start`) — bare shortcut forms like `pnpm dev` are skipped because they
collide with each package manager's own built-in CLI verbs, and a naive rule
would create false positives. That's why AgentReady stayed silent on
AIReady's README documenting `pnpm dev` with no root `dev` script.

### Scope

- Add `commands.reference.shortcut-script` alongside the existing
  `commands.reference.npm-script`/`make-target`/`package-manager-mismatch`
  finding kinds.
- Maintain an allowlist of real package-manager built-ins per manager (npm,
  yarn, pnpm, bun) so a shortcut form is only flagged when it unambiguously
  cannot be a built-in.
- Respect the existing workspace-qualifier exclusion
  (`isWorkspaceQualified`/`WORKSPACE_FLAG_PATTERN`) — a shortcut scoped to a
  workspace package is not a root-script claim.
- Emit `error` only where resolution is unambiguous; emit a lower-confidence
  `info` finding where it is not, and say in the finding detail why the
  command was interpreted as a package script rather than merely "looks
  wrong."

### Acceptance criteria

- `pnpm dev` with no root `dev` script and no built-in `dev` verb is flagged.
- Real built-ins (`npm ci`, `yarn why`, `pnpm add`, ...) are never flagged.
- A workspace-qualified shortcut is never flagged.
- Unit tests cover at least one true positive and one built-in per supported
  package manager.

## Issue 3: Composite finding for automatic-hook code execution

Labels: `enhancement`, `detector`, `safety`, `v0.4`

**Status: delivered.** `safety.agent-hook.executes-repository-code`
(`detectHookExecutionRisks` in `lib/repo-readiness/detectors/safety-signals.ts`)
reads `.claude/settings.json`/`.claude/settings.local.json`, flags a hook
configured on a session-lifecycle event (`SessionStart`, `SessionEnd`,
`PreCompact`, `Notification` — events that fire without any explicit user
action, as opposed to `PreToolUse`/`PostToolUse`/`UserPromptSubmit`/`Stop`,
which are tied to the user already actively driving the session) whose
command matches a package-manager install pattern. Default `warning`; the
`enterprise` policy pack escalates to `error`.

### Problem

`capability-surfaces.ts` already assigns a `high` risk tier to a non-empty
Claude Code `hooks` settings block, and `safety-signals.ts` already flags
install-lifecycle `package.json` scripts — but as two separate, generic
findings. Neither names the specific composite risk AIReady's `SessionStart`
hook has: an agent checks out an untrusted branch, a session hook fires
automatically with no explicit user action, the hook runs dependency
installation, and installation can execute lifecycle scripts controlled by
that same untrusted branch.

### Scope

- Add `safety.agent-hook.executes-repository-code`: fires when a detected
  hook surface (`.claude/settings.json`'s `hooks` block, a
  `.claude/hooks/*` script, or equivalent surfaces for other agent tools) both
  (a) runs on an automatic/session-start-style event rather than one requiring
  explicit user invocation, and (b) its command invokes a package-manager
  install/lifecycle command or an equivalent local build/generate step.
- Evidence should include: hook file, hook event, the command executed,
  whether that command can invoke repository lifecycle scripts, and whether
  the event is automatic or user-invoked.
- Default severity `warning`; candidate for `enterprise` policy-pack
  escalation to `error` alongside the existing
  `safety.capability.high-risk`/`safety.install-hook` escalations in
  `lib/repo-readiness/checks/policy-packs.ts`.

### Acceptance criteria

- AIReady-shaped fixture (SessionStart hook running `pnpm install`) produces
  the new finding, distinct from (in addition to, not replacing) the existing
  generic hook/install-hook findings.
- A hook that only runs on explicit user invocation (e.g. a slash command) is
  not flagged by this rule.
- A hook that runs automatically but does not invoke install/lifecycle
  commands is not flagged by this rule.

## Issue 4: Autonomy-stage metadata and autonomy-envelope report section

Labels: `enhancement`, `report-contract`, `v0.4`

**Status: delivered.** `AgentStage`/`AutonomyStageResult`/`AutonomyStatus`
(`lib/repo-readiness/core/types.ts`); every `RuleDoc` in
`checks/catalog.ts` carries `affectedStages`; `calculateAutonomyEnvelope`
derives `report.autonomyEnvelope` (additive, added to
`reportContract.experimentalFields`, schema version stays
`local-readiness/v2`); rendered as a "### Autonomy envelope" section in
`formatScanMarkdown` and a not-ready/blocked-only `Autonomy:` line in
`formatScanSummary`.

### Problem

A single aggregate score cannot communicate "ready to edit and open a PR, not
ready to merge or deploy autonomously." AgentReady already has per-dimension
category scores; this is the next layer.

### Scope

- Define `AgentStage = 'orient' | 'bootstrap' | 'navigate' | 'edit' | 'verify'
  | 'review' | 'merge' | 'deploy'` (same enum used by the calibration schema's
  `affectedStage` from Issue 1) and attach it to each rule's catalog entry in
  `lib/repo-readiness/checks/catalog.ts`.
- Derive an autonomy-envelope summary (per stage: ready / not-yet-ready /
  blocked, with the driving findings) as an additive, optional report field —
  following the same additive-spine approach as `docs/adr/0000-canonical-report-contract-and-rollout.md`.
  Raw findings and `summary.score` stay unchanged; this is a derived view, not
  a new scoring mechanism.
- Surface it in the markdown/console reporters.

### Acceptance criteria

- Every existing rule in the catalog has at least one `affectedStage`.
- JSON Schema and report-contract docs are updated for the new optional field,
  with a version bump matching the existing `reportContract.experimentalFields`
  rollout convention.
- A repository with only `merge`/`deploy`-stage findings still reports
  `orient`/`edit`/`verify` as ready.

## Issue 5: "Not verified from repository contents" reporting

Labels: `enhancement`, `report-contract`, `v0.4`

**Status: delivered.** `NOT_VERIFIED_EXTERNAL_CONTROLS`
(`lib/repo-readiness/reporters/not-verified.ts`) is a fixed list, rendered as
a "### Not verified from repository contents" section in
`formatScanMarkdown` and a compact `Not verified from repository contents:
...` line in `formatScanSummary`. Left out of `scan --format json`'s schema
(consistent with keeping that contract policy/presentation-independent) —
the list is static and not report-derived, so JSON consumers that need it can
import the same constant directly.

### Problem

Branch protection, required status checks, required CODEOWNER review, and
environment approval rules are not reliably observable through a local
filesystem scan — but a report that says nothing about them can read as
"nothing to worry about" rather than "not checked." AgentReady's local-first,
non-networked scan is a deliberate trust guarantee (see README "Design
guarantees") and should stay that way; this issue is about being explicit
about its edges, not about adding network calls to the core scan.

### Scope

- Add a fixed "Not verified from repository contents" list to the
  markdown/console report (branch protection, required status checks,
  required CODEOWNER review, environment approval rules, production secret
  scopes, deployment permission boundaries).
- Keep it a static, always-present section for the default (offline) scan —
  no attempt to infer these from local files.
- Design it so a future opt-in GitHub Action enrichment mode (see roadmap
  candidate below, not scoped for v0.4) could later replace list entries with
  real answers, without changing the section's shape.

### Acceptance criteria

- Section appears in every default scan's markdown/console output.
- Core `scan --format json` schema is unaffected (or the section is added
  there too as an explicit, versioned additive field — pick one during
  implementation and document the choice).

## Issue 6: Protected-path CODEOWNERS coverage

Labels: `enhancement`, `detector`, `governance`, `v0.4`

**Status: delivered**, with one deliberate deviation: findings are `info` by
default (matching every other candidate-expectation finding in the
deterministic core, e.g. `docs.codeowners.coverage-gap`) rather than
policy-gated invisibility, because `PolicyPack.adjust` can only change an
existing finding's severity, not conjure a new one from evidence the core
never computed — see Issue 7 for the same constraint. The `enterprise` policy
pack escalates both `governance.codeowners.protected-path-gap` and
`governance.codeowners.single-owner-risk` to `warning`.
`detectProtectedPathCoverage` (`lib/repo-readiness/detectors/governance.ts`)
takes a `protectedPaths` parameter (defaults to `DEFAULT_PROTECTED_PATHS`,
also exported) rather than a config-file option — same fixed-default-instead-
of-config-plumbing choice as Issue 7.

### Problem

`detectCodeownersCoverageGaps` (`lib/repo-readiness/detectors/governance.ts`)
already flags top-level directories with sustained recent commit activity
that no CODEOWNERS pattern covers — but it's activity-derived, not
risk-derived. AIReady's CODEOWNERS covers a fixtures path with sustained
activity, assigned to one individual, while `.github/**`, `.claude/**`, and
other structurally high-risk paths have no dedicated owner and may not show
up as "active" by commit-count alone.

### Scope

- Add a configurable protected-path list (defaults: `.github/**`,
  `.claude/**`, `AGENTS.md`, `CLAUDE.md`, `**/auth/**`, `**/security/**`,
  `**/migrations/**`, `**/schema.*`, `**/deploy/**`, `**/infra/**`,
  `**/prompts/**`, `**/fixtures/*cassette*`).
- `governance.codeowners.protected-path-gap`: a protected path with no
  covering CODEOWNERS pattern.
- `governance.codeowners.single-owner-risk`: a protected path covered by
  exactly one individual owner, with no team or backup.
- Keep the existing activity-derived `docs.codeowners.coverage-gap` unchanged;
  this is an additional, risk-derived signal, primarily for the `enterprise`
  policy pack (`lib/repo-readiness/checks/policy-packs.ts`).

### Acceptance criteria

- AIReady-shaped fixture (CODEOWNERS covering only a cassette-fixture glob,
  one owner) produces both new findings for `.claude/**`/`.github/**`-style
  gaps.
- A repository with a team-owned (`@org/team`) protected path is not flagged
  by `single-owner-risk`.
- Findings are opt-in via policy (not part of the `default` pack's severity),
  matching how other enterprise-only expectations are scoped in
  `docs/product/policy-packs.md`.

## Issue 7: `policyOptions.requirePortableInstructionEntrypoint`

Labels: `enhancement`, `policy`, `v0.4`

**Status: delivered**, scoped down from the original ask. The generic
`policyOptions` config-file/CLI-flag plumbing described below did not ship —
`PolicyPack.adjust` has no mechanism to synthesize a new finding from
evidence the deterministic core doesn't already compute a finding for, and
building that mechanism (a `synthesize` hook threaded through `applyPolicy`,
`evaluateScanGate`/`evaluateDiffGate`, the CLI, and the Action) was judged too
invasive for this milestone. Instead, `instructions.portable-entrypoint.missing`
ships as an always-computed `info`-severity rule (fixed default:
`AGENTS.md`) — consistent with how every other "enterprise candidate
expectation" in this codebase already works (compute at low severity by
default, let the `enterprise` policy pack escalate). The `enterprise` pack
escalates it to `warning`. The generic `policyOptions` shape remains a
candidate; see `docs/product/policy-packs.md`.

### Problem

`docs/product/policy-packs.md` already documents a planned `policyOptions`
config shape (`minScore`, `requireInstructionEntrypoint`,
`allowIntentionalFixtureData`) that has not shipped — thresholds are
currently fixed per pack. The default `instructions.missing` check correctly
treats any recognized instruction surface as sufficient (AgentReady
deliberately avoids assuming one filename is canonical); AIReady centers
exclusively on a vendor-specific `CLAUDE.md` with no `AGENTS.md`-style
portable entrypoint, which is a real portability gap for multi-agent or
enterprise use without being a default-policy defect.

### Scope

- Ship the `policyOptions` config shape from `docs/product/policy-packs.md`
  (currently "not yet delivered" per that doc's Status section).
- Add `requirePortableInstructionEntrypoint` +
  `portableInstructionPaths` (default: `["AGENTS.md"]`).
- New finding: `instructions.portable-entrypoint.missing`, gated behind this
  option (off by default, expected on for an `enterprise`-style policy
  configuration).

### Acceptance criteria

- Default pack behavior is unchanged with `policyOptions` unset.
- `enterprise` (or a repo-level config) can turn the option on and see the new
  finding when only a vendor-specific entrypoint exists.
- Raw finding IDs from the deterministic core stay unaffected; this is a
  policy-layer addition, not a change to `instructions.missing`.

## Candidates not scoped for v0.4

Kept here so they aren't lost, but they need more design work before an issue
draft is actionable:

- **Command-contract drift** (`pnpm validate` describing lint+typecheck+test
  while the manifest also runs `knip`): an optional LLM-assisted analysis
  comparing a manifest script against nearby documentation's description of
  it, distinct from the existing structural
  `instruction-contradictions.ts` (which compares instruction files against
  each other, not against the manifest).
- **Developer-guidance drift beyond instruction files** (stale task
  playbooks, outdated check/plugin taxonomies): broadening the same
  optional-analysis idea to a bounded set of README/CONTRIBUTING/playbook/
  architecture docs, with the report explaining which documents were
  selected and why.
- **Optional GitHub governance enrichment** (`enrich-github-governance: true`
  Action input) to answer the Issue 5 "not verified" list from inside CI,
  where a token already exists — kept strictly separate from the local,
  non-networked core scan contract, the same way policy interpretation is
  kept separate from raw findings today.
