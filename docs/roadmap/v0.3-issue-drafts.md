# v0.3 Issue Drafts

These are ready-to-open GitHub issue drafts for the next milestone. They are
kept in-tree so the roadmap is visible even before issues are created.

## Milestone: v0.3 — Policy and evidence calibration

Goal: turn AgentReady from a strong scanner into a calibrated product: clear
positioning, policy packs, instruction-quality analysis, and benchmark evidence
against real agent friction.

Done when:

- policy-pack selection is available in CLI, Action, and report metadata
- at least one non-default policy pack changes severity in a tested way
- instruction-quality findings are documented with examples
- benchmark summary covers a small mixed corpus and records false positives /
  false negatives
- README links to sample reports and explains how AgentReady differs from CI,
  lint, Scorecard, and security scanners

## Issue 1: Add policy-pack interface and `--policy` plumbing

Labels: `enhancement`, `policy`, `v0.3`

### Problem

The core scanner is intentionally broad and descriptive, but teams need stronger
or domain-specific gates without changing raw evidence.

### Scope

- Define a typed `PolicyPack` interface.
- Add a no-op `default` policy over today's behavior.
- Add `--policy <name>` to `scan` and `diff`.
- Add `policy` input to the GitHub Action.
- Include selected policy and effective thresholds in JSON/markdown/SARIF report
  metadata.

### Acceptance criteria

- Existing scans without `--policy` are unchanged except for metadata.
- Unknown policy names fail with a clear CLI/Action error.
- Unit tests cover CLI parsing, report metadata, and default no-op behavior.
- Docs show the policy selection shape.

## Issue 2: Implement first meaningful policy pack

Labels: `enhancement`, `policy`, `v0.3`

### Problem

Users need to see how policy packs make AgentReady more useful than a static
score.

### Scope

Choose one pack for the first slice:

- `enterprise`: stricter gates for explicit instructions, CI/local command
  parity, dangerous scripts, and capability surfaces.
- or `ml-scientific`: tuned treatment for sample data, notebooks, generated
  bindings, and mixed Python/C++ build systems.

### Acceptance criteria

- The selected pack adjusts severity for at least three rule families.
- Raw finding IDs and evidence remain unchanged.
- Reports explain every severity adjustment with a policy reason.
- Diff gates can fail on policy-adjusted severity.

## Issue 3: Publish sample AgentReady reports

Labels: `documentation`, `examples`, `v0.3`

### Problem

Users should understand AgentReady output before installing or running it.

### Scope

- Add one high-readiness sample report.
- Add one improvement-plan sample report with realistic findings.
- Link both from the README.
- Keep samples small and manually stable; they are product examples, not raw
  dogfood artifacts.

### Acceptance criteria

- README links render correctly.
- Samples include score, key evidence, findings, and next actions.
- Samples avoid claiming a published npm package until it exists.

## Issue 4: Instruction-quality analysis release story

Labels: `enhancement`, `analyze`, `docs`, `v0.3`

### Problem

Many repositories already have instruction files. The differentiator is whether
those instructions are useful, scoped, fresh, and non-contradictory.

### Scope

- Document deterministic versus optional LLM responsibilities.
- Add examples of actionable versus weak instruction guidance.
- Ensure analyze-layer output clearly explains instruction-quality adjustments.
- Add dogfood notes showing at least two real instruction-quality findings.

### Acceptance criteria

- `scan` remains deterministic and model-free.
- `analyze` output includes itemized rationale and remediation.
- Docs show how host-delegated MCP analysis avoids AgentReady owning secrets.

## Issue 5: Real-agent benchmark summary

Labels: `evaluation`, `benchmark`, `v0.3`

### Problem

Readiness scores need calibration against real coding-agent behavior.

### Scope

- Select a small mixed corpus of public repositories.
- Run deterministic scans and summarize tracked findings.
- Run bounded tasks with at least two coding agents where feasible.
- Record true positives, false positives, and missing signals.

### Acceptance criteria

- Raw artifacts stay outside the tracked repo or are sanitized.
- Tracked summary includes corpus, commands, finding categories, and decisions.
- At least three scanner changes or policy decisions are linked back to observed
  agent friction.

## Issue 6: Repository metadata and first-impression polish

Labels: `documentation`, `release`, `v0.3`

### Problem

The GitHub repository should communicate the product before users read the full
README.

### Scope

- Set GitHub About description: `Local-first repository readiness scanner for AI coding agents`.
- Add topics such as `ai`, `coding-agents`, `readiness`, `cli`, `github-action`,
  `agents-md`.
- Remove or correct stale homepage links.
- Add a short README quick-look section with sample report links.

### Acceptance criteria

- Repository metadata matches README positioning.
- README does not claim npm availability before publish.
- New users can see what AgentReady does in under 30 seconds.
